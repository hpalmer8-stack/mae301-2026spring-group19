## EchoGuard – MVP Report
1. Executive Summary:
EchoGuard is an AI-based engine diagnostic tool that analyzes short audio recordings of engine idle and determines whether the engine is operating normally or exhibiting abnormal behavior. The system uses acoustic feature extraction combined with machine learning classification to produce a binary output (healthy vs abnormal) along with a confidence score. The MVP demonstrates a complete pipeline from audio input to prediction, supported by a simple user interface that allows users to upload engine audio and receive immediate feedback. This approach provides a fast and accessible method for early detection of potential engine issues.
2. User & Use Case:
The primary users of EchoGuard are vehicle owners, DIY and professional mechanics, and fleet managers seeking a quick, accessible way to assess engine health without specialized diagnostic tools. For example, a truck owner who hears an unusual sound while the engine is idling can record a short audio clip and upload it to the system. EchoGuard then analyzes the recording and returns a classification indicating whether the engine is likely operating normally or abnormally. This helps users make informed decisions about whether further inspection or maintenance is necessary.
3. System Design:
The system consists of four main components: audio input, preprocessing, feature extraction, and classification. User-provided audio is first standardized through preprocessing steps such as mono conversion, resampling, and clipping to a fixed duration. The processed audio is then transformed into numerical features, including MFCCs, spectral centroid, RMS energy, and zero-crossing rate. These features are passed into a trained machine learning model, which outputs a classification and confidence score. The result is presented to the user through a simple interface, completing the end-to-end diagnostic workflow.
4. Data:
The model was trained using publicly available engine audio datasets sourced from Kaggle, including datasets focused on engine condition and automotive diagnostics. These datasets provide labeled examples of both normal and abnormal engine sounds. All audio files were standardized to a consistent format and split into training and testing sets to support model development and evaluation. In addition, real-world engine recordings from team members’ vehicles were collected and used as external validation data to assess model performance outside the training dataset.
5. Models:
The MVP uses supervised machine learning models trained on extracted acoustic features. Baseline models, such as logistic regression and random forests, were implemented to distinguish between healthy and abnormal engine sounds. Feature extraction focused on commonly used audio descriptors, including MFCCs, spectral centroid, RMS energy, and zero-crossing rate. In addition to the classification model, our nanoGPT implementation generates natural-language explanations of the predictions, providing users with more interpretable diagnostic feedback.
6. Evaluation:
Model performance was evaluated using standard classification metrics, including accuracy, precision, recall, and F1 score. A confusion matrix was used to visualize the performance of the predictions across the two classes. In addition to quantitative evaluation, qualitative testing was performed using real engine recordings from team members’ vehicles. These tests demonstrated that the model performs well on structured dataset inputs, while also highlighting some variability when applied to real-world audio conditions.
7. Limitations & Risks:
The current system is limited to binary classification and does not identify specific engine faults. Performance can also be affected by variations in audio quality, background noise, and differences between the training data and real-world recordings. These factors introduce some uncertainty in predictions, particularly when evaluating audio outside the dataset distribution. Additionally, the model’s effectiveness depends on the diversity and quality of the training data, which may not fully represent all engine types and conditions.
8. Next Steps:
With additional development time, we would expand the model to support multi-class classification for specific engine faults, such as misfires or valve-related issues. We also plan to improve model accuracy and consistency across varying audio conditions by expanding the dataset and refining preprocessing techniques. On the product side, we would enhance the user interface and incorporate more detailed explanations and recommendations, creating a more complete and user-friendly diagnostic tool.
